# ペルソナ & ユースケース定義

> ペルソナとユースケースを定義することで、機能要求・DB設計・ダッシュボード設計の判断基準とする。

---

## ロール定義

> 認証機能は実装スコープ外。画面は管理者ロールで構築し、ロールは概念として定義する。詳細は README を参照。

| ロール | 説明 | 操作権限 |
|--------|------|---------|
| **管理者（Admin）** | システム全体を管理する | 全操作（CRUD・削除含む） |
| **一般ユーザー** | ログの閲覧・検索・分析を行う | 閲覧・検索・作成・編集のみ（削除不可） |

---

## ペルソナ

### Persona A — 運用保守担当（Kenji）
- **役割:** システムの日常監視・障害対応を担当
- **技術レベル:** コードは読まない。ダッシュボードやツールを通じてシステム状態を把握する
- **関心:** 「今、システムに異常はないか」を定期的に確認したい
- **課題:** ログが大量にあり、問題の兆候を見つけるのに時間がかかる

### Persona B — バックエンド開発者（Yuki）
- **役割:** サービスの開発・バグ修正を担当
- **技術レベル:** コードを読み書きする。ターミナルでの作業も可能だが、ブラウザから素早く確認したい
- **関心:** 「このエラーはいつから・どこで発生しているか」を素早く特定したい
- **課題:** 障害報告を受けてから原因特定までに時間がかかる

### Persona C — チームリード（Hanako）
- **役割:** 複数サービスを跨いだ品質管理・意思決定を担当
- **技術レベル:** 技術的な背景はあるが、詳細調査より俯瞰的な把握を優先する
- **関心:** 「システム全体の健全性トレンド」を定期的に把握したい
- **課題:** 複数のメンバーから口頭で状況を聞かないと全体像が掴めない

## ユースケース

### UC-01 — 異常の早期検知（Persona A）
**シナリオ:** Kenji が朝の定期確認でダッシュボードを開く

1. ダッシュボードを開き、更新ボタンで最新データを取得する
2. severity 別のサマリーカードで ERROR / CRITICAL 件数を確認する
3. 前日と比べてエラーが急増していれば、時系列チャートで増加し始めた時刻を特定する
4. フィルタパネルで source を絞り込み、どのサービス・コンポーネントが原因かを特定する
5. ログ一覧ページで該当時間帯のログを検索し、メッセージの詳細を確認する

**必要な機能:** 更新ボタン / サマリーカード / 時系列チャート / source フィルタ / ログ検索

---

### UC-02 — エラー原因の調査（Persona B）
**シナリオ:** Yuki が「API がエラーを返している」という報告を受けて調査する

1. ログ一覧ページで severity=ERROR・日付範囲=直近1時間に絞り込む
2. source でサービス名を指定してさらに絞り込む
3. 該当ログの詳細ページを開き、メッセージ本文・timestamp を確認する

**必要な機能:** severity / source / 日付範囲フィルタ / ログ詳細表示

---

### UC-03 — 健全性トレンドの把握（Persona C）
**シナリオ:** Hanako が週次レビューでシステムの状態を確認する

1. ダッシュボードで過去1週間の日付範囲を設定する
2. 時系列チャートで ERROR 件数の週次推移を確認する
3. severity 分布ヒストグラムで INFO / WARNING / ERROR / CRITICAL の割合を確認する
4. 問題があれば担当者に確認する。必要に応じて CSV でエクスポートして共有する

**必要な機能:** 日付範囲フィルタ / 時系列チャート / severity 分布ヒストグラム / CSV エクスポート

---

### UC-04 — ログの手動登録（Persona A / B）
**シナリオ:** 自動収集されていないログを手動で記録する

1. ログ作成ページを開く
2. timestamp / message / severity / source を入力して保存する

**必要な機能:** ログ作成フォーム / 入力バリデーション

---

### UC-05 — ログの管理（管理者）
**シナリオ:** 手動登録した誤ったログを修正・削除する

1. ログ一覧または詳細ページから対象ログを特定する
2. 手動登録ログの内容に誤りがあれば編集する
3. 不要なログを削除する（管理者権限）

> **注:** ログは証跡としての信頼性を保つため、削除は管理者のみが実行できる操作とする。また、一定期間経過したログの自動削除（リテンションポリシー）も運用上の選択肢として想定する（`docs/backlog.md` 参照）。

**必要な機能:** ログ編集 / ログ削除（管理者）

## ユースケースから導かれる設計方針

### ダッシュボードで表示すべき指標

| 指標 | 対応ユースケース | 優先度 |
|------|----------------|--------|
| severity 別サマリーカード（ERROR数・CRITICAL数など） | UC-01 / UC-03 | 必須 |
| 時系列ログ件数トレンドチャート | UC-01 / UC-03 | 必須 |
| 手動更新ボタン | UC-01 | 必須 |
| severity 分布ヒストグラム | UC-03 | ボーナス |
| CSV エクスポート | UC-03 | ボーナス |

### DBカラムへの示唆

| カラム | 根拠 |
|--------|------|
| `timestamp` | UC-01/02 で時刻による絞り込みが必須 |
| `message` | UC-02 で本文確認が必須 |
| `severity` | UC-01/02/03 で絞り込み・集計の軸 |
| `source` | UC-01/02 で発生源の特定に必須 |

### 将来のユースケース（バックログ）

- **AI ディベートエンジン:** ログデータ・要求・要件を入力に OSS LLM 2台がディベートしてインサイトを提示する（`docs/backlog.md` 参照）
- **リアルタイム更新:** WebSocket による自動 Push（現状は手動更新ボタンで代替）
- **認証・権限管理:** JWT ベースの認証でロールを実際に分離する
