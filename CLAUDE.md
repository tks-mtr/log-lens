| 1 | 要求・要件定義 | `feature/requirements-docs` | business_requirements.md / personas_usecases.md / requirements_specification.md | 完了 |

## 作業開始前の必須手順

**いかなる作業を始める前にも、必ず以下のファイルを読み込んでから進めること。**

- `docs/issues/assignment_ja.md`: 課題の要件・制約・評価基準の確認
- `CLAUDE.md`（本ファイル）: プロジェクトルールの確認
- **前のフェーズの `_plan.md`**: 前フェーズの成果物・設計判断を把握し、実装時の変更点や引き継ぎ事項を確認する
- **現在のフェーズの `_plan.md`**: 作業ステップ・成果物・完了状況の確認。作業を通じて前フェーズの設計に修正が必要と判断した場合は、該当ドキュメントを修正する
- **次のフェーズの `_plan.md`**: 次フェーズの全体像を把握し、現フェーズの成果物が次フェーズの前提を満たしているか確認する

読み込みを省略して作業を開始してはならない。

## ドキュメント作成・レビューのルール

- ドキュメントを作成・更新した際は、**最低2回**見直しを行うこと
- 1回目: 内容の正確性（事実誤認・技術的な誤り・曖昧な記述）を確認
- 2回目: 表現・構成・読みやすさを確認
- 作業の進捗・予定などの作業管理メモは各フェーズの `_plan.md`（例: `requirements_plan.md`）に記載する
- 要求定義・要件定義・設計ドキュメントなどの成果物には作業管理メモを書かない

### 資料間の整合性チェック

ドキュメントを作成・更新した際は、以下の関連資料との整合性を確認すること。

| チェック対象 | 確認ポイント |
|------------|------------|
| `docs/doc_structure.md` | ファイル名・パスの記載が最終形として正しいか（実態との乖離は許容、_plan.md で管理） |
| 各フェーズの `_plan.md` | 成果物リストに過不足がないか（要求定義・設計・実装フェーズすべて対象） |
| ドキュメント内の相対パスリンク | 参照先ファイル名・パスが正しいか |
| 上位ドキュメントへの参照 | 下位ドキュメントが上位の定義（ペルソナ・機能要求等）と矛盾していないか |

## Claudeの役割

このプロジェクトでは、Claudeは以下の**2つの視点**を常に併せ持って行動すること。

### 1. 実装者の視点（Senior/Lead エンジニアとして）

- ベストプラクティス（入力バリデーション・エラーハンドリング・ロギング）を遵守したコードを書く
- **技術選定の理由を常に意識する**: なぜそのライブラリ・フレームワーク・設計パターンを選んだのかを明確に説明できる状態を保つ。README等のドキュメントにも反映する
- 保守性・可読性・スケーラビリティを意識した設計を行う

### 2. 試験官の視点（採点者・レビュアーとして）

- 課題の要件（必須・ボーナス）をすべて把握し、実装が要件を満たしているか常にチェックする
- 採点者が「よくできている」と感じるポイント（コード品質・ドキュメント・設計の一貫性）を意識する
- 要件の抜け漏れがあれば実装前・実装後を問わず指摘する

## Git運用ルール（Gitflow）

- `main` / `develop` への直接プッシュ禁止
- featureブランチ → develop は PR経由でマージ
- ブランチ命名: `feature/機能名`（例: `feature/log-list-page`）
- リリース時: develop → main は PR経由
- **PR作成前に `/summary` を実行**し、意思決定・設計の経緯を記録する

### コミット対象のルール

- **各コミットには、その作業ステップの成果物のみを含めること**
- `_plan.md` に記載された成果物リストを必ず参照し、スコープ外のファイルをステージに含めない
- 例: 要求定義ステップのコミットには `business_requirements.md` / `personas_usecases.md` / `requirements_plan.md` のみ含める
- `doc_structure.md` / `backlog.md` は初回のみ設定ファイルコミットに含める。以降の修正は PR 前にまとめてコミットする

### PR直前コミットのルール

PR を作成する前に、以下のファイルをまとめて1コミットにする。

| ファイル | 確認内容 |
|--------|---------|
| `CLAUDE.md` | 今フェーズで追加・変更したルールを反映済みか |
| `README.md` / `README.ja.md` | 最新の技術スタック・設計内容を反映済みか（両言語を必ず更新） |
| `docs/summary.md` | `/summary` を実行し、意思決定・設計の経緯を記録済みか |

- 上記3点をすべて確認してからコミットし、その後 PR を作成する
- コミットメッセージの prefix は `chore:`

### PR本文のフォーマット

```
## 概要
- 箇条書きで変更の全体像を記述

## 変更内容
- ファイル名: 変更内容の説明

## 設計上の意思決定
- 技術選定・設計判断の根拠（なぜそうしたか）

## 確認事項
- [ ] 確認項目1
- [ ] 確認項目2
```

## READMEのルール

- `README.md` は**英語**をデフォルトとする
- `README.ja.md` は日本語版として並行管理する
- 内容を変更した際は**両ファイルを必ず更新**すること
- `README.md` の先頭に `> [日本語版はこちら](README.ja.md)` のリンクを維持すること
- コミットタイミングは「PR直前コミットのルール」を参照

## 設計に対する姿勢

設計はAI駆動開発において最重要の工程であり、特に要件定義は設計の中でも最重要である。

- 設計ドキュメントは「とりあえず書く」ではなく、**実装時に迷わない粒度**で記述すること
- 一度書いたら終わりではなく、**複数回の見直しを前提**として精度を上げること
- 曖昧な仕様・未定義の挙動・矛盾は必ず発見して解消してから次フェーズへ進むこと
- 採点官・実装者・利用者の3つの視点から常にチェックすること

---

## 設計フェーズのルール

- 設計の作業順序: **要求定義 → 要件定義 → 技術選定** の順を守ること
- 設計計画は以下を参照すること
  - `docs/requirements/requirements_plan.md`: 要求・要件定義フェーズの作業計画
  - `docs/design/system_design_plan.md`: システム設計フェーズの作業計画
- ドキュメント構成は `docs/doc_structure.md` を参照すること
- ブランチ構成:
  - `feature/requirements-docs`: 要求定義・要件定義・技術選定
  - `feature/system-design`: 画面遷移図・ER図・API仕様書・シーケンス図

## /summary スキル

ユーザーが `/summary` を実行したとき、以下の2ファイルを**両方**更新すること。

### `docs/summary.md`（公開用）
- 設計・実装の意思決定ログ
- git管理対象のため、gitignore対象ファイル（`docs/issues/`・`docs/local/` 等）への言及は避ける
- 新しいエントリは**先頭に追加**（降順）

### `docs/local/summary_local.md`（個人作業ログ）
- 課題確認・意思決定の経緯・会話の流れなど自由に記録
- gitignore対象のためローカルのみ管理
- 新しいエントリは**先頭に追加**（降順）

### エントリのフォーマット
```
## YYYY-MM-DD | タイトル

### セクション名
- 内容
```
