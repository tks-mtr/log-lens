# Git運用ルール

## Gitflow

- `main` / `develop` への直接プッシュ禁止
- featureブランチ → develop は PR経由でマージ
- ブランチ命名: `feature/機能名`（例: `feature/log-list-page`）
- リリース時: develop → main は PR経由
- **PR作成前に `/summary` を実行**し、意思決定・設計の経緯を記録する

## コミット対象のルール

- **各コミットには、その作業ステップの成果物のみを含めること**
- `_plan.md` に記載された成果物リストを必ず参照し、スコープ外のファイルをステージに含めない
- 例: 要求定義ステップのコミットには `business_requirements.md` / `personas_usecases.md` / `requirements_plan.md` のみ含める
- `doc_structure.md` / `backlog.md` は初回のみ設定ファイルコミットに含める。以降の修正は PR 前にまとめてコミットする
- **コミット履歴は `amend` / `fixup` を積極的に使ってまとめること**
  - 直前のコミットへの追加: `git commit --amend`
  - 過去のコミットへの追加: `git commit --fixup=<hash>` → `git rebase -i --autosquash HEAD~N`（N は対象コミットの距離 + 1。対象が `HEAD~4` なら `HEAD~5` を指定）

## PR直前コミットのルール

PR を作成する前に、以下のファイルをまとめて1コミットにする。

| ファイル | 確認内容 |
|--------|---------|
| `CLAUDE.md` | 今フェーズで追加・変更したルールを反映済みか・フェーズ状態を「完了」に更新したか |
| `SKILL.md` | 今フェーズで追加・変更したスキルを反映済みか |
| `.vscode/settings.json` | IDE設定（インタープリタパス等）を反映済みか |
| `README.md` / `README.ja.md` | 最新の技術スタック・設計内容を反映済みか（両言語を必ず更新） |
| `docs/summary.md` | `/summary` を実行し、意思決定・設計の経緯を記録済みか |
| 現フェーズの `_plan.md` | 全ステップの完了状態・完了条件のチェックボックスを更新済みか |

- 上記5点をすべて確認してからコミットし、その後 PR を作成する
- コミットメッセージの prefix は `chore:`

## PR本文のフォーマット

```
## 概要
- 箇条書きで変更の全体像を記述

## 変更内容
- ファイル名: 変更内容の説明

## 設計上の意思決定
- 技術選定・設計判断の根拠（なぜそうしたか）

## 確認事項
- [ ] 確認項目1
- [ ] 確認項目2
```
