# コードレビューのルール

コードレビューを行う前に、以下の設計書を必ず読み込むこと。設計上の意図を把握せずにレビューを行ってはならない。

| 読み込むファイル | 確認ポイント |
|----------------|------------|
| `docs/requirements/requirements_specification.md` | API仕様・バリデーション・エラーハンドリング・テスト方針など「どう作るか」の定義 |
| 現在のフェーズの `_plan.md` | 実装の注意点・設計上の意思決定（例: BOM付きCSV、Service層の責務など） |

レビュー結果を報告する際は、各指摘が「設計書に反している」のか「設計書の範囲外のコード品質問題」なのかを明示すること。

## テストコードのレビュー確認項目

テストコードのレビュー時は以下も確認すること。

- 正常系・異常系・境界値がすべて含まれているか
- Repository 層は異常系（空結果・存在しないID等）が含まれているか
- アサーションが具体的か（`is not None` だけで終わっていないか）
- テスト名が `test_<対象>_<条件>_<期待結果>` 形式になっているか
